转移指令机器码计算

在软件分析中，需要经常计算转移指令机器码或修改指定的代码，掌握其原理和技巧显得非常有必要。根据转移的距离不同，转移指令可分为以下三类：
短转移（Short Jump）：无条件转移和条件转移的机器码均为2字节，转移的范围是 -128~127字节
长转移（Long Jump）：无条件转移的机器码为5字节，条件转移的机器码为6字节。
这是因为：条件转移要用2字节表示其转移类型（如je、jg、jns等），其他4字节表示转移偏移量，而无条件转移仅用1字节就可以表示转移类型（jmp），其他4字节表示转移偏移
子程序调用指令（Call）：Call指令分为两类。
①一类是我们平时常见的类似于长转移，（例：CALL 00401100)
②另一类调用的参数涉及寄存器、栈等值，比较复杂。（例：CALL dword ptr[eax+2]）

条件转移指令的转移范围是16位模式遗留下来的，CPU开发人员只给目的地址分配了1字节，所以其转移的长度限制在255字节之内。

Windows下的异常处理

1 异常处理的基本概念
Intel公司在386开始的IA-32家族处理器开始，引入了中断（interrupt）和异常（exception）的概念。
中断是由外部硬件设备或者 异步事件产生的
异常是由于内部事件产生的，可分为故障、陷阱和终止。
故障和陷阱是可恢复的；终止类异常是不可恢复的，如果发生了这种异常，系统就必须重启
1.1异常列表
异常是应用程序正常执行过程中发生的不正常事件。
由CPU引发的异常称为硬件异常，由操作系统或者应用程序引发的异常称为软件异常。
常见的异常如下

2 异常处理的基本过程
Windows正常启动后，将运行在保护模式下，当有中断或异常发生时，CPU会通过中断描述符表（Interrupt Descriptor Table,IDT）来寻找处理函数。IDT是CPU和操作系统交接中断和异常的关口。
（1）IDT
IDT是一张位于物理内存中的线性表，共256项。在32位模式下每个IDT项的长度为8字节，在64位模式下每个IDT项的长度为16字节。操作系统在启动阶段会初始化这个表，系统中的每个CPU都有一份IDT的拷贝。
IDT的位置和长度由CPU的IDTR寄存器描述的。IDTR寄存器共有48位，其中高32位是表的基址，低16位是表的长度。可以使用SIDT和LIDT指令来读写该寄存器，但是LIDT是特权指令，只能在Ring0特权级下运行。

TEB
TEB指线程环境块，该结构体包含进程中运行线程的各种信息，进程中的每个线程都对应一个TEB结构体。不同的OS中TEB结构体形态略微不同
TEB重要成员
在用户模式调试中有两个成员起着重要的作用

ProcessEnvironmentBlock
ProcessEnvironmentBlock是指向PEB结构体的指针（Process Environment Block，PEB 进程环境块，每个进程对应一个PEB结构体）
NtTib
TEB结构体的第一个成员为_NT_TIB结构体（TIB Thread Information Block 线程信息块）

第一个成员ExceptionLIst 指向_EXCEPTION_REGISTRATION_RECORD结构体组成的链表，用于Windows OS的SEH
SEH(“Structured Exception Handling”),即结构化异常处理·是(windows)操作系统提供给程序设计者的强有力的处理程序错误或异常的武器。
来自百度百科
最后一个Self成员是_NT_TIB结构体的自引用指针，也是TEB结构体的指针（因为TEB结构体的第一个成员就是 _NT_TIB结构体）
TEB访问方法
使用OS提供的API来访问TEB
Ntdll.NtCurrentTeb()
Ntdll.NtCurrentTeb()API用于返回当前线程的TEB结构体的地址。
TEB结构体的地址，与FS段寄存器所指的段内存的基地址是一样的。也就是说TEB结构体与FS段寄存器有着某种联系
FS段寄存器用来指示当前线程的TEB结构体，但是并非直接指向TEB结构体的地址，它持有SDT索引，该索引持有实际TEB的地址
SDT位于内核内存区域，其地址存储在特殊的寄存器GDTR中（GDTR 全局描述符表寄存器 Global Descriptor Table Resiger）

FS:[0x18]=TEB起始地址
FS：[0x18]=TEB.NtTib.Self=address of TIB =address of TEB=FS : 0
*FS:0 是段内存的起始地址，FS寄存器指向一个段描述符，
而该描述符又指向段内存的起始地址

调用函数时，会先将函数起始地址（ESP）保存到EBP，并维持在函数内部，这样无论ESP怎么变化，以EBP的值为基准能够安全访问到相关函数的局部变量、参数、返回地址。借助战阵技术管理函数调用时，无论函数有多深、多复杂，调用栈都能得到很好的管理与维护。

PE文件格式
PE文件是windows OS 下使用的可执行文件格式。一个.exe文件运行时 如何加载到内存、从何处运行、运行中需要的DLL又哪些、需要多大的栈/堆内存等大量信息，都以结构体形式存储在PE头中。
基本结构
1.从DOS头到节区头是PE头部分。文件中使用偏移（offset），内存中使用VA (Virtual Address）来表示位置，文件加载到内存时节区大小、位置等都会发生变化。文件的内容一般可分为代码（.text）、数据（.data）、资源（.rsrc） 节，分别保存。

2.各节区头定义了各节区在文件或内存中的大小、位置、属性等。文件/内存中节区的起始位置应该在各文件/内存最小单位的倍数位置上，空白区域用NULL填充。
3.VA指进程虚拟内存的绝对地址，RVA指从基准位置开始的相对地址,PE头内部信息大多以RVA 形式存在
RVA+ImageBase=VA

函数调用约定

Calling Convention 既“函数调用约定”，他是对函数调用时如何传递参数的一种约定。函数调用约定有一下三种：

cdecl(c语言默认)

cdecl是C语言使用的方式，调用者main（）函数直接清理其压入栈内的函数参数，
cdecl方式的好处在于，它可以像C语言的printf（）函数一样，
向被调用者传递长度可变的参数。这种长度可变的参数在其他调用约定中很难实现。

stdcall

stdcall方式常用于Win32 API 该方式由被调用者清理栈。
若想在C语言中使用stdcall方式编译源码，只要使用_stdcall关键字即可

stdcall方式的好处在于，被调用者函数内部存在着栈清理代码，
与每次调用函数时都要用ADD ESP,XXX 命令的方式相比，代码尺寸小

fastcall

fastcall方式与stdcall方式基本类似，但是该方式通常会使用寄存器（而非栈内存）去传递那些需要传递给函数的部分参数。
其优点在于可以实现对函数的快速调用（寄存器在CPU内部），但是有时需要额外的系统开销来管理ECX,EDX寄存器。若调用函数前ECX与EDX中存有重要数据，那么使用前就必须先备份
